# ADD Sync - Ecosystem Consistency Validator

> **LANG:** Respond in user's native language (detect from input). Tech terms always in English.

Scans the ADD framework ecosystem (`framwork/.codeadd/`), validates consistency between commands and skills, detects drift in README/web/SVGs, and regenerates the ecosystem map from real data.

**Run before `/add.release` to ensure ecosystem is consistent.**

---

## Spec

```json
{"output":{"map":"framwork/.codeadd/skills/code-addiction-ecosystem/SKILL.md"},"modes":{"full":"STEP 1-7","dry-run":"STEP 1-5 + STEP 7 (no writes)","map-only":"STEP 1-2 + STEP 6 + STEP 7"}}
```

## ⛔⛔⛔ MANDATORY SEQUENTIAL EXECUTION ⛔⛔⛔

**STEPS IN ORDER:**
```
STEP 1: Scan ecosystem          → Read ALL source files in framwork/.codeadd/
STEP 2: Build cross-reference   → Map commands <-> skills dependencies
STEP 3: Detect inconsistencies  → Orphans, missing context, gaps
STEP 4: Detect README drift     → Compare README.md vs code changes since last release
STEP 5: Detect web/SVG drift    → Compare web pages + SVGs vs scanned ecosystem data
STEP 6: Regenerate ecosystem map → Rewrite SKILL.md from scan data
STEP 7: Report                  → Console output with all findings
```

```
IF SCAN NOT COMPLETE:
  ⛔ DO NOT USE: Write on any file
  ⛔ DO NOT USE: Edit on any file
  ⛔ DO NOT: Generate report or map
  ✅ DO: Complete scan of all directories first

IF CROSS-REFERENCE NOT DONE:
  ⛔ DO NOT USE: Write on ecosystem map
  ⛔ DO NOT: Report inconsistencies
  ✅ DO: Complete dependency analysis first

IF MODE IS --dry-run:
  ⛔ DO NOT USE: Write on ANY file
  ⛔ DO NOT USE: Edit on ANY file
  ⛔ DO NOT: Modify ecosystem map
  ✅ DO: Report only (console output)

ALWAYS:
  ⛔ DO NOT USE: Write on provider directories (framwork/.claude/, framwork/.agent/, framwork/.agents/)
  ⛔ DO NOT USE: Edit on provider directories (framwork/.claude/, framwork/.agent/, framwork/.agents/)
  ⛔ DO NOT USE: Write on command files (framwork/.codeadd/commands/)
  ⛔ DO NOT USE: Edit on command files (framwork/.codeadd/commands/)
  ⛔ DO NOT USE: Write on skill content files (except ecosystem map)
  ⛔ DO NOT USE: Write on scripts (framwork/.codeadd/scripts/)
  ⛔ DO NOT USE: Write on README.md (only report drift)
  ⛔ DO NOT USE: Write on web pages or SVGs (only report drift)
```

## STEP 1: Scan Ecosystem (MANDATORY FIRST)

### 1.1 Scan Command Directories

Read ALL files in the source of truth:

```
framwork/.codeadd/commands/*.md    → CodeADD commands (SOURCE OF TRUTH)
```

For EACH command file in `.codeadd/commands/`:

1. Extract filename (e.g., `add.new.md`)
2. Read first 5 lines to get name and description
3. Search file content for skill references:
   - Grep for `.codeadd/skills/` paths
   - Grep for `SKILL.md` mentions
   - Grep for skill names in `skills:` JSON fields or "MANDATORY" sections
4. Search for `owner.md` reference
5. Search for `feature-status.sh` reference
6. Search for `product.md` reference
7. Search for `code-addiction-ecosystem` reference
8. Search for `--yolo` support

Store as:

```json
{
  "name": "add.new",
  "description": "Feature Discovery & Documentation",
  "skills_referenced": ["feature-discovery", "feature-specification", "documentation-style"],
  "has_owner_md": true,
  "has_feature_status": false,
  "has_product_md": false,
  "has_ecosystem_next_steps": false,
  "has_yolo": false
}
```

### 1.2 Scan Skill Directories

Read ALL SKILL.md files in the source of truth:

```
framwork/.codeadd/skills/*/SKILL.md    → CodeADD skills (SOURCE OF TRUTH)
```

For EACH skill in `.codeadd/skills/`:

1. Extract directory name (e.g., `backend-development`)
2. Read first 10 lines for name and description from frontmatter
3. Search for `## When to Use` or `Usada por` to identify which commands use it

Store as:

```json
{
  "name": "backend-development",
  "description": "Patterns NestJS, Clean Arch, DI, DTOs",
  "referenced_by_commands": []
}
```

### 1.3 Scan Scripts

List and read first 5 lines of each file in:

```
framwork/.codeadd/scripts/*
```

### 1.4 Scan Templates

```
framwork/.codeadd/templates/*
```

**Mark scan as COMPLETE.**

## STEP 2: Build Cross-Reference Map

### 2.1 Commands → Skills

For EACH command from STEP 1.1, populate `skills_referenced` by matching:
- Explicit paths: `.codeadd/skills/[name]/SKILL.md`
- Skill names mentioned in Spec JSON: `"skills":{"mandatory":"..."}`
- Skill names in text: `Load [skill-name] skill`

### 2.2 Skills → Commands (reverse mapping)

For EACH skill from STEP 1.2, populate `referenced_by_commands`:
- Which commands reference this skill (from 2.1)?
- If `referenced_by_commands` is empty → mark as **ORPHAN**

### 2.3 Compare with Current Ecosystem Map

Read existing `framwork/.codeadd/skills/code-addiction-ecosystem/SKILL.md`.

Identify DRIFT:
- Commands in map but NOT on disk → **REMOVED** (map is stale)
- Commands on disk but NOT in map → **MISSING** (map incomplete)
- Skills in map but NOT on disk → **REMOVED**
- Skills on disk but NOT in map → **MISSING**
- Skill references in map that differ from scan → **CHANGED**

**Mark cross-reference as COMPLETE.**

## STEP 3: Detect Inconsistencies

### 3.1 Orphan Skills

Skills with `referenced_by_commands == []`:

```
[ORPHAN] skill-name: not referenced by any command
```

### 3.2 Missing Context Loading

For EACH command in `.codeadd/commands/`, check:

| Check | Expected | If Missing |
|-------|----------|------------|
| `owner.md` | ALL commands | `[WARN] add-X: does not load owner.md` |
| `feature-status.sh` | Commands that operate on features* | `[WARN] add-X: does not use feature-status.sh` |
| `product.md` | `add.new` only | `[WARN] add.new: does not load product.md` |
| `code-addiction-ecosystem` at completion | ALL commands with completion step | `[WARN] add-X: no next-steps suggestion` |

*Feature-operating commands: `add.new`, `add.design`, `add.plan`, `add.build`, `add.autopilot`, `add.review`, `add.done`, `add.test`, `add.hotfix`, `add.brainstorm`, `add.ux`, `add.pr`

### 3.3 Phantom References

Skills referenced by commands but NOT found on disk:

```
[ERROR] add-X references skill "phantom-skill" but framwork/.codeadd/skills/phantom-skill/SKILL.md does not exist
```

### 3.4 Summary Counts

```
Total commands: X
Total skills: Y
Total scripts: Z
Orphan skills: N
Missing context loading: N
Phantom references: N
```

## STEP 4: Detect README Drift

### 4.1 Get Last Release Tag

```bash
git fetch --tags
git tag --sort=-v:refname | head -1
```

**CRITICAL:** Without `git fetch --tags`, remote tags are invisible locally → incorrectly assumes first release.

Store as `LAST_TAG` (e.g., `v0.2.22`).

### 4.2 Get Changed Files Since Last Release

Diff `LAST_TAG..HEAD` to get list of changed files. Store as `CHANGED_FILES`.

### 4.3 Read README.md

Read `README.md` (root).

### 4.4 Cross-Check README vs Changes

For EACH category below, check if `CHANGED_FILES` includes relevant modifications that should be reflected in `README.md`:

| What changed | README section to check | Rule |
|---|---|---|
| New command in `.codeadd/commands/` | "The Development Trail" table | If new command not in table → `[DRIFT]` |
| Removed command | "The Development Trail" table | If removed command still in table → `[DRIFT]` |
| Command renamed or description changed | "The Development Trail" table | If table row outdated → `[DRIFT]` |
| New provider directory in `framwork/` | "What gets installed" list | If new provider not listed → `[DRIFT]` |
| Removed provider | "What gets installed" list | If removed provider still listed → `[DRIFT]` |
| New flow added or changed | "Choose your flow" section | If flows outdated → `[DRIFT]` |
| CLI commands changed in `cli/` | "Quickstart" code block | If CLI usage outdated → `[DRIFT]` |
| Roadmap items delivered | "Roadmap" section | If delivered item still listed as roadmap → `[DRIFT]` |
| Repo structure changed (new top-level dirs) | "Repository structure" section | If structure outdated → `[DRIFT]` |

Store findings as:

```json
{
  "readme_drifts": [
    {
      "section": "The Development Trail",
      "issue": "New command /add.audit not listed in trail table",
      "severity": "WARN"
    }
  ]
}
```

## STEP 5: Detect Web/SVG Drift

### 5.1 Read Web Pages

```
Read web/src/pages/index.astro
Read web/src/pages/docs.astro
```

### 5.2 Read SVG Assets

```
Read web/public/commands.svg
Read web/public/flows.svg
Read web/public/flowchart.svg
```

### 5.3 Cross-Check Web Pages vs Scanned Data

Compare the SCANNED ecosystem data (from STEP 1) against what is hardcoded in the web pages:

#### docs.astro — Commands Section

For EACH command found in STEP 1 scan:
- Check if it appears in the Commands grid in `docs.astro`
- Check if its description matches

Classify:
- `[WEB-MISSING]` — command exists on disk but NOT in docs.astro
- `[WEB-EXTRA]` — command listed in docs.astro but NOT on disk
- `[WEB-OUTDATED]` — command exists but description/category differs

#### docs.astro — Skills Section

For EACH skill found in STEP 1 scan:
- Check if it appears in the skills array in `docs.astro`
- Check if its description matches

Classify: `[WEB-MISSING]` / `[WEB-EXTRA]` / `[WEB-OUTDATED]`

#### docs.astro — Flows Section

Compare flows in `docs.astro` against flows listed in README.md and ecosystem map:
- Are all flows represented?
- Do flow steps match current command names?

Classify: `[WEB-OUTDATED]` if flow steps changed

#### docs.astro — Providers Section

Compare providers table in `docs.astro` against provider directories found on disk in `framwork/`:
- `[WEB-MISSING]` — provider dir exists but not in table
- `[WEB-EXTRA]` — provider in table but no dir on disk

#### index.astro — Content

Check if `index.astro` references outdated information:
- Provider list in "Runs everywhere" section matches disk
- Install command is current
- Flow descriptions match current flows

### 5.4 Cross-Check SVGs vs Current State

For EACH SVG in `web/public/`:

| SVG | Should reflect | Check |
|-----|---------------|-------|
| `commands.svg` | All commands from STEP 1 scan | Commands in SVG match commands on disk |
| `flows.svg` | All flows (Complete, Standard, Lean, Autonomous, Emergency, Exploration) | Flow steps match current command names |
| `flowchart.svg` | Development trail diagram | Steps match current trail in README |

For each SVG, read its content and:
1. Extract all command names mentioned (grep for `/add.` patterns)
2. Compare against scanned commands from STEP 1
3. Flag commands missing from SVG or extra commands in SVG that no longer exist

Classify:
- `[SVG-OUTDATED]` — SVG references commands/flows that changed
- `[SVG-MISSING]` — SVG doesn't include new commands/flows
- `[SVG-OK]` — SVG matches current state

Store findings as:

```json
{
  "web_drifts": [
    {
      "file": "web/src/pages/docs.astro",
      "section": "Commands",
      "type": "WEB-MISSING",
      "detail": "Command /add.audit not in Commands grid"
    }
  ],
  "svg_drifts": [
    {
      "file": "web/public/commands.svg",
      "type": "SVG-OUTDATED",
      "detail": "SVG still references /add.update which was removed"
    }
  ]
}
```

## Rules

ALWAYS:
- Scan ALL source directories in framwork/.codeadd/ before any analysis
- Cross-reference commands with skills bidirectionally
- Show diff before writing ecosystem map
- Preserve Main Flows section from existing map
- Include date in Last Updated with action description
- Run `git fetch --tags` BEFORE reading tags (remote tags may be invisible locally)
- Compare README.md sections against code changes since last release tag
- Read `web/src/pages/*.astro` and compare against scanned ecosystem data
- Read `web/public/*.svg` and verify command/flow references match disk
- Report all README, web page, and SVG drift items in STEP 7

NEVER:
- Modify command files (only ecosystem map)
- Modify skill content files (only ecosystem map entry)
- Modify README.md, web pages, or SVGs (only report drift)
- Delete any files (only create and update)
- Write anything in `--dry-run` mode
- Write to any provider directory (framwork/.claude/, framwork/.agent/, framwork/.agents/) — deploy artifacts
- Assume map is correct without comparing to disk
- Skip `git fetch --tags` before reading tag list
